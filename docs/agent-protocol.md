# Протокол роботи агента

Цей файл описує робочий процес для агентів, які допомагають вести репозиторій `engineering-history`.

## Основний принцип

Не переписувати історію заднім числом. Додавати нові свідчення, уточнення й виправлення окремими, зрозумілими змінами.

## Формат роботи

1. Користувач розповідає подію, людину, технічне рішення або принцип.
2. Агент відокремлює факти від припущень і художніх реконструкцій.
3. Один самостійний сюжет або одна логічна зміна оформлюється як окремий файл або мінімальний patch.
4. Один patch має відповідати одному commit.
5. Агент створює готовий unified diff у файлі `.patch`.
6. Разом із patch агент дає точні команди перевірки, застосування, перегляду й commit.
7. Користувач сам перевіряє, застосовує, комітить і відправляє зміни.

## Обов'язкова перевірка

Перед застосуванням:

```bash
git status
git apply --check <patch-file>
```

Після застосування:

```bash
git apply <patch-file>
git diff --check
git status
git diff
```

Після перевірки тексту:

```bash
git add <changed-files>
git commit -m "<short meaningful message>"
git push
```

## Стиль текстів

- Зберігати голос і лексику автора.
- Не додавати пафосу, героїзації або вигаданих деталей.
- Не маскувати невпевненість: припущення позначати як припущення.
- Технічні деталі фіксувати точно, навіть якщо вони здаються дрібними.
- Цитати залишати дослівними, якщо автор підтвердив формулювання.
- Не перетворювати живу пам'ять на офіційну біографію.
- Краще короткий завершений фрагмент, ніж великий незакінчений розділ.

## Структура змін

Рекомендовані каталоги:

- `projects/` — інженерні проєкти;
- `people/` — люди й особисті спогади;
- `principles/` — робочі та життєві принципи;
- `docs/` — правила ведення репозиторію та інструкції агентам.

Назви файлів мають бути короткими, зрозумілими та стабільними.

## Сигнали в розмові

- `PATCH CANDIDATE` — матеріал варто зберегти, але patch ще не створюється.
- `роби патч` або `зроби патч` — створити готовий `.patch` і команди.
- `не той бранч` — нічого не створювати й не переносити до репозиторію.
- `+++` — підтвердження напряму або результату; це не автоматична команда на commit.

## Межі агента

Агент не повинен стверджувати, що має доступ до GitHub або локального репозиторію, якщо такого доступу немає.

За замовчуванням агент:

- створює patch-файл;
- дає команди;
- не виконує `commit` і `push`;
- не вигадує структуру наявних файлів, якщо зміна залежить від неї;
- для нових незалежних файлів може створювати мінімальний patch без зміни інших документів.

## Безперервність

Якщо агент зміниться або попередня розмова стане недоступною, наступний агент має прочитати цей файл перед роботою.

Мета процесу проста: не допустити, щоб важливий досвід зник лише тому, що його носій, редактор або агент більше недоступний.
